# 合理使用MT与MD

---

### MT对比MD

|                            | /MT (静态链接)                                  | **/MD** (动态链接)                                  |
| :------------------------- | :---------------------------------------------- | --------------------------------------------------- |
| **CRT 链接方式**           | 静态链接到 .exe/.dll 中                         | 动态链接到 `msvcp140.dll` 和 `ucrtbase.dll`         |
| **文件大小**               | 较大（CRT 代码被复制进每个模块）                | 较小（共享系统 DLL）                                |
| **部署依赖**               | 无需运行时库 DLL                                | 需要确保目标系统有 VC++ 运行时（或通过安装包分发）  |
| **多个 DLL 共享 CRT 状态** | ❌ 每个模块有独立的 CRT 堆，跨模块释放内存会崩溃 | ✅ 所有模块共享同一个 CRT 堆，跨模块 new/delete 安全 |
| **安全更新**               | 需要重新编译所有模块才能获得 CRT 安全补丁       | 系统更新 VC++ 运行时即可，无需重编译                |
| **调试**                   | 调试需要额外的静态调试库                        | 调试依赖动态调试 DLL（通常已安装）                  |

### 使用场景

#### 推荐使用MD动态链接

1. 希望**减小每个模块的体积** 
2. 项目会生成**多个 DLL**，并且这些 DLL 之间需要互相传递 C++ 对象（如 `std::string`、`std::vector`）或在一个 DLL 中 `new` 在另一个 DLL 中 `delete` 
3. 部署时可以通过安装包附带 VC++ 运行时
4. 需要使用 **MFC** 或 **ATL**（它们通常也是动态链接的）

#### 推荐使用MT静态链接

1. 项目是一个**独立的单文件 .exe**，没有多个 DLL 互相传递 CRT 对象
2. 需要**简化部署**（复制一个 .exe 就能运行，不需要额外安装运行时）
3. 对文件大小不敏感（现代 CRT 静态增加几百 KB，通常可接受）

### 项目中具体场景

teamtalk客户端中正常编译一个modules.dll的动态库，该动态库下包含很多其他的子模块，

1. **多个 DLL/EXE 之间可能共享数据**（比如主 EXE 和 modules.dll 都使用 jsoncpp，如果它们分别静态链接 CRT，则各自有自己的堆，跨模块传递 json::Value 对象可能导致崩溃）
2. **项目已经使用了 MFC**（从之前的 `afxwin.h` 可以看出），而 MFC 通常也是动态链接的，默认需要 `/MD` 
3. **对于整个解决方案，所有项目必须使用相同的运行时库设置（所有用 `/MT`，或所有用 `/MD`），否则会收到 LNK2038 不匹配错误**
