# 系统服务目录结构设计

---

### 推荐结构

1. connection：不写业务规则，只做收包、鉴权入口，然后转发到service\handler
2. service\handler：不直接操作底层连接细节，通过服务层接口
3. domain：尽量无网络依赖，方便测试与复用
4. common：不放带有状态的全局对象，状态对象放到service或registry类
5. 先按照边界拆文件，然后再做类的拆分

```
server/msg_server/
  main.cpp
  CMakeLists.txt
  connection/              # 连接与会话层（网络事件、连接对象、路由入口）
    msg_conn.h/.cpp
    route_serv_conn.h/.cpp
    file_serv_conn.h/.cpp
    attach_data.h/.cpp
  handlers/                # 协议命令处理（按命令域拆）
    login/
    single_chat/
    group_chat/
    file/
    buddy/
    ...                    # 每个目录里是对应 handler
  domain/                  # 纯业务模型与规则（尽量不依赖网络）
    user/
      im_user.h/.cpp
    group/
    session/
    message/
  services/                # 用例编排层（连接层与domain/db之间的协调）
    message_service.h/.cpp
    group_service.h/.cpp
    online_service.h/.cpp
  storage/                 # 数据访问层（DB/缓存）
    db_proxy_client/
    cache/
    repository/
  common/                  # 公共工具（无业务状态）
    json/
    utils/
    constants/
    error/
  protocol/                # 协议定义/编解码适配（如 pb 封装）
    pdu/
    converters/
  config/
    msg_server.conf
```
