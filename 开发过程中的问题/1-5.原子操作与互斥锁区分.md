# 原子操作与互斥锁区分

---

在某些场景下使用原子操作，要比直接使用互斥锁，程序性能将得到更大的提升。

### 历史代码

```cpp
class CRefObject {
 public:
  CRefObject();
  virtual ~CRefObject();

  void SetLock(CLock* lock) { m_lock = lock; }

  void AddRef();
  void ReleaseRef();

 private:
  int m_refCount;
  CLock* m_lock;
};
```

```cpp
CRefObject::CRefObject() {
  m_lock = NULL;
  m_refCount = 1;
}

CRefObject::~CRefObject() {}

void CRefObject::AddRef() {
  if (m_lock) {
    m_lock->lock();
    m_refCount++;
    m_lock->unlock();
  } else {
    m_refCount++;
  }
}

void CRefObject::ReleaseRef() {
  if (m_lock) {
    m_lock->lock();
    m_refCount--;
    if (m_refCount == 0) {
      delete this;
      return;
    }
    m_lock->unlock();
  } else {
    m_refCount--;
    if (m_refCount == 0)
      delete this;
  }
}
```

### 原子操作优化后代码

```cpp
// CRefObject.h
class CRefObject {
 public:
  CRefObject();
  virtual ~CRefObject();

  void AddRef();
  void ReleaseRef();
  
 private:
  std::atomic<int> m_refCount;
};
```

```cpp
// CRefObject.cpp
CRefObject::CRefObject() : m_refCount(1) {}

CRefObject::~CRefObject() {}

void CRefObject::AddRef() {
  m_refCount.fetch_add(1, std::memory_order_relaxed);
}

void CRefObject::ReleaseRef() {
  int oldCount = m_refCount.fetch_sub(1, std::memory_order_acq_rel);
  if (oldCount == 1) {
    delete this;
  }
}
```

### 优化方案

1. `SetLock` 方法是否有必要？
   - 没有必要，而且这是一个糟糕的设计
   - 允许外部设置锁会导致生命周期管理混乱（谁负责删除这个mutex？）
   - 这种设计违反了单一职责原则，让对象的线程安全依赖外部状态
   - 改用 `std::mutex m_lock;` 替代指针
2. 使用 `std::atomic<int>` 替代 `std::mutex` 
   - 性能更好：原子操作比互斥锁快得多（无需系统调用）标准的原子操作，线程安全且高效，
   - 性能差距：atomic约1-5ns，mutex约100ns（差20-100倍）
   - 内存更小：`atomic<int>`通常4字节，额外的mutex对象
   - `std::atomic` 是专为这种场景设计的（如 `std::shared_ptr` 内部实现）
   - 无死锁风险：消除了原代码中的重复加锁bug
3. 使用正确的内存序
   - `AddRef`：使用 `memory_order_relaxed`（只需保证原子性）
   - `ReleaseRef`：使用 `memory_order_acq_rel`（确保delete前所有操作完成）

| 方案           | 内存开销 | 操作耗时 |
| :------------- | :------- | :------- |
| mutex成员      | ~40字节  | ~100ns   |
| atomic（推荐） | 4字节    | ~1-5ns   |

### 原子操作相关

原子操作是不可被线程调度机制打断的操作，一旦开始就运行到结束。在多线程环境下，对共享数据的原子操作可以避免数据竞争（data race），无需使用互斥锁即可保证操作的完整性。C++11 标准库提供了 `<atomic>` 头文件和 `std::atomic<T>` 模板类，将原子操作封装成易用的类型。

原子操作为多线程编程提供了无锁的基础同步手段。C++ 的 `std::atomic` 易用且高效，适合计数器、标志位、自旋锁等简单场景。对于复杂数据结构，仍推荐结合互斥锁或无锁数据结构库。

#### 常用原子类型

1. **基本整数类型**：`std::atomic<int>`, `std::atomic<long>`, `std::atomic<bool>` 
2. **指针类型**：`std::atomic<int*>` 
3. **自定义类型**：需满足 `std::is_trivially_copyable<T>::value`（如 POD 类型）

#### 核心成员函数

| 操作                                              | 说明                |
| :------------------------------------------------ | :------------------ |
| `load()`                                          | 原子读取值          |
| `store(val)`                                      | 原子写入值          |
| `exchange(val)`                                   | 交换并返回旧值      |
| `fetch_add(n)` / `fetch_sub(n)`                   | 原子加/减，返回旧值 |
| `++` / `--`                                       | 前置/后置自增自减   |
| `compare_exchange_weak/strong(expected, desired)` | CAS（比较并交换）   |

#### 内存顺序

原子操作可指定内存顺序，控制不同线程间操作的可见性：

| 内存顺序                   | 适用操作                   | 语义                     |
| :------------------------- | :------------------------- | :----------------------- |
| `memory_order_relaxed`     | 任意                       | 无同步，仅原子性         |
| `memory_order_acquire`     | load / 读-改-写（读部分）  | 获取                     |
| `memory_order_release`     | store / 读-改-写（写部分） | 释放                     |
| **`memory_order_acq_rel`** | **读-改-写**               | **获取+释放**            |
| `memory_order_seq_cst`     | 任意                       | 获取+释放+全局顺序一致性 |

`memory_order_acq_rel` 是 C++ 原子操作中一种**读-改-写**操作专用的内存顺序，它结合了**获取**（acquire）和**释放**（release）两种语义：

1. **获取语义**：保证当前线程中，该原子操作之后的读写操作不会被重排序到该操作之前。并且使其他线程中对该原子变量的**释放**操作对该线程可见。
2. **释放语义**：保证当前线程中，该原子操作之前的读写操作不会被重排序到该操作之后。并且当前线程对原子变量的写入会**释放**到其他线程（使其在后续获取操作中可见）。
3. 典型使用场景：用于**需要同时读旧值并写新值**的原子操作，例如 `fetch_add`、`exchange`、`compare_exchange_strong/weak` 
4. `memory_order_acq_rel` 等价于对读部分施加 acquire，对写部分施加 release。

**`memory_order_acquire`**：用于 **load** 或 **读-改-写操作的读部分**：

作用：保证该操作**之后**的读写不会跑到它前面去。相当于“进门”操作。

**`memory_order_release`**：用于 **store** 或 **读-改-写操作的写部分**：

作用：保证该操作**之前**的读写不会跑到它后面去。相当于“出门”操作。

例如 `fetch_add`、`exchange`、`compare_exchange`，它们会**读取旧值，再写入新值**。

1. 如果只用 `acquire`，那么写操作没有 release 语义，可能导致其他线程看不到这次写入的同步效果。
2. 如果只用 `release`，那么读操作没有 acquire 语义，可能读到过时的值且允许后续操作重排序到前面。
3. 因此 **`acq_rel`** 同时施加两种约束：
   - 对**读**部分施加 acquire 语义（防止后续操作重排到前面）
   - 对**写**部分施加 release 语义（防止前面的操作重排到后面）

#### 实际使用实例

##### 原子计数器

```cpp
#include <atomic>
#include <thread>
#include <vector>
#include <iostream>

std::atomic<int> counter{0};  // 原子整数

void worker() {
    for (int i = 0; i < 1000; ++i) {
        counter.fetch_add(1, std::memory_order_relaxed);  // 原子自增
        // 或者直接用 ++counter;
    }
}

int main() {
    std::vector<std::thread> threads;
    for (int i = 0; i < 10; ++i) {
        threads.emplace_back(worker);
    }
    for (auto& t : threads) t.join();
    std::cout << "Final counter: " << counter.load() << std::endl;  // 输出 10000
    return 0;
}
```

##### 自旋锁（利用 CAS 实现简单锁）

```cpp
class SpinLock {
    std::atomic<bool> flag{false};
public:
    void lock() {
        while (flag.exchange(true, std::memory_order_acquire)) {
            // 自旋等待
        }
    }
    void unlock() {
        flag.store(false, std::memory_order_release);
    }
};
```

##### 单次初始化

单次初始化（`std::call_once` 更常用，但也可用原子实现）

```cpp
std::atomic<bool> initialized{false};
void init() { /* 初始化代码 */ }

void use() {
    bool expected = false;
    if (initialized.compare_exchange_strong(expected, true)) {
        init();  // 仅第一个到达的线程执行
    }
}
```

#### 注意事项

1. **原子操作不等于线程安全**：复合操作（如先读后写依赖原值的操作）仍需使用 CAS 循环或锁。
2. **性能**：原子操作比锁轻量，但过度使用或使用强内存顺序仍可能造成性能下降。
3. **自定义类型**：`std::atomic<LargeStruct>` 可能不支持（若无锁实现），编译时会报错，建议使用 `std::shared_ptr` 或锁。
